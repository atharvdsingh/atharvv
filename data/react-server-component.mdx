# React Server Components: The Mental Model Shift

**March 25, 2025** · 6 min read

If you're still thinking about React the way you did in 2022, you're leaving performance on the table. Server Components changed everything — but most tutorials explain the *what* without the *why*. Let me fix that.

## The Old Way Was Broken

Here's how we used to build a blog page:

```tsx
"use client";

export default function BlogPage() {
  const [posts, setPosts] = useState([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch("/api/posts")
      .then((res) => res.json())
      .then((data) => {
        setPosts(data);
        setLoading(false);
      });
  }, []);

  if (loading) return <Skeleton />;
  return <PostList posts={posts} />;
}
```

What's wrong with this?

1. **The user sees nothing** until JavaScript loads, parses, and executes
2. **Two round trips** — first the page, then the API call
3. **The entire component ships to the browser** even though it just displays data
4. SEO crawlers see a blank page

## The Server Component Way

With Server Components, data fetching happens *where the data lives* — on the server:

```tsx
// No "use client" — this runs on the server
import { prisma } from "@/lib/db";

export default async function BlogPage() {
  const posts = await prisma.post.findMany({
    orderBy: { createdAt: "desc" },
    take: 10,
    select: {
      id: true,
      title: true,
      slug: true,
      excerpt: true,
      createdAt: true,
    },
  });

  return <PostList posts={posts} />;
}
```

What changed:

- **Zero client-side JavaScript** for this component
- **One round trip** — HTML ships fully rendered
- **Direct database access** — no API layer needed
- **SEO-friendly** out of the box

## The Mental Model: Server vs Client

Here's the rule I follow:

| Question | Server Component | Client Component |
|---|---|---|
| Does it fetch data? | ✅ | ❌ |
| Does it use `useState`? | ❌ | ✅ |
| Does it use `useEffect`? | ❌ | ✅ |
| Does it handle click events? | ❌ | ✅ |
| Does it access browser APIs? | ❌ | ✅ |
| Does it display static content? | ✅ | ❌ |

> Think of Server Components as the **skeleton** and Client Components as the **muscles**. The skeleton gives structure; the muscles add interactivity.

## The Composition Pattern

The trick is to push `"use client"` as far down the tree as possible:

```tsx
// page.tsx — Server Component (no directive)
import { prisma } from "@/lib/db";
import { LikeButton } from "./like-button";

export default async function PostPage({
  params,
}: {
  params: { slug: string };
}) {
  const post = await prisma.post.findUnique({
    where: { slug: params.slug },
  });

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.content}</p>

      {/* Only THIS part ships JS to the browser */}
      <LikeButton postId={post.id} />
    </article>
  );
}
```

```tsx
// like-button.tsx — Client Component
"use client";

import { useState } from "react";

export function LikeButton({ postId }: { postId: string }) {
  const [liked, setLiked] = useState(false);

  const handleLike = async () => {
    setLiked(true);
    await fetch(`/api/posts/${postId}/like`, {
      method: "POST",
    });
  };

  return (
    <button onClick={handleLike}>
      {liked ? "❤️" : "🤍"} Like
    </button>
  );
}
```

The entire post content renders on the server. Only the 200-byte like button ships JavaScript.

## Common Mistakes

### 1. Making everything a Client Component

```tsx
// ❌ Don't slap "use client" on your layout
"use client";

export default function Layout({ children }) {
  return <div className="max-w-3xl mx-auto">{children}</div>;
}
```

This component has **zero interactivity**. It should be a Server Component.

### 2. Passing non-serializable props

```tsx
// ❌ Functions can't cross the server→client boundary
<ClientComponent onClick={() => console.log("hi")} />
```

```tsx
// ✅ Pass data, not functions
<ClientComponent postId={post.id} />
```

### 3. Using `useEffect` for data fetching

If you're writing `useEffect` + `fetch` in a Next.js 14+ app, stop. Use a Server Component or React Query instead. Every `useEffect` fetch is a waterfall waiting to happen.

## Performance Comparison

Here's what I measured on a real project after migrating:

| Metric | Before (Client) | After (Server) | Improvement |
|---|---|---|---|
| **First Contentful Paint** | 2.1s | 0.8s | 62% faster |
| **JavaScript Bundle** | 187 KB | 42 KB | 78% smaller |
| **Time to Interactive** | 3.4s | 1.2s | 65% faster |
| **Lighthouse Score** | 71 | 96 | +25 points |

These aren't theoretical numbers. This is what happens when you stop shipping unnecessary JavaScript.

## The Takeaway

Server Components aren't just an optimization — they're a **fundamentally better way to think about React**. Data stays on the server. Interactivity lives on the client. The boundary between them is explicit and clean.

Start with Server Components by default. Only add `"use client"` when you need interactivity. Your users — and your Lighthouse scores — will thank you.
